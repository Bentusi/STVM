## 关键设计决策

### 1. 指令集简化
**决策：** 从原计划的40+指令简化到28个核心指令

**理由：**
- 减少解释器分支预测失败
- 更容易实现和理解
- 指令正交性更好
- 为JIT优化打基础

**示例：** 
- 合并 `OP_PUSH_INT`, `OP_PUSH_REAL` → `OP_PUSH`（类型由常量池决定）
- 合并 `OP_LOAD_LOCAL`, `OP_LOAD_GLOBAL` → `OP_LOAD`（用flags区分）

### 2. 错误处理简化
**决策：** 移除独立的错误管理子系统

**理由：**
- 错误码作为返回值更清晰
- 减少全局状态
- 错误信息存储在上下文中更自然
- 简化API

**影响：**
- error.h只定义错误码和辅助宏
- 无error.c实现文件
- 各模块结构体包含error_code和error_msg字段

### 3. 内存池设计
**决策：** 使用多级内存池加专用池

**配置：**
```c
POOL_SMALL:  64字节  × 1024块  (小对象)
POOL_MEDIUM: 512字节 × 512块   (中对象)
POOL_LARGE:  4KB     × 128块   (大对象)
POOL_AST:    128字节 × 2048块  (AST节点)
POOL_SYMBOL: 256字节 × 1024块  (符号表)
POOL_STRING: 64字节  × 2048块  (字符串)
```

**优势：**
- 减少malloc/free开销
- 提高内存局部性
- 便于统计和调试
- 支持快速重置

### 4. AST节点设计
**决策：** 使用联合体 + 链表指针

**结构：**
```c
struct ASTNode {
    ASTNodeType type;
    SourceLocation location;
    ASTNodeData data;  // 联合体
    ASTNode* next;     // 链表
    TypeInfo* resolved_type;
};
```

**优势：**
- 节省内存（联合体共享空间）
- 便于构建节点列表
- 支持类型推导结果存储

### 5. 符号表实现
**决策：** 哈希表 + 作用域链

**实现细节：**
- 256桶哈希表（每个作用域）
- 冲突用链表解决
- 作用域通过parent指针连接
- 支持限定名查找

**查找策略：**
1. 检查是否是限定名（含`.`）
2. 限定名只在全局作用域查找
3. 普通名从当前作用域向上查找

## 遗留问题

### 1. 类型系统内存管理
**问题：** type_info_free递归释放导致double free

**原因：** 
- 数组/函数类型包含子类型指针
- 释放时不清楚所有权归属

**建议解决方案：**
1. 使用引用计数
2. 明确所有权规则（谁创建谁释放）
3. 使用arena分配器统一管理

### 2. 内存池边界检查
**问题：** 当前池满时回退到malloc，释放时判断不够准确

**改进方向：**
- 在MemoryBlock中添加magic number
- 更严格的边界检查
- 考虑使用arena分配器

### 3. AST节点所有权
**当前：** 不够明确，容易导致泄漏或double free

**改进：**
- 文档化所有权规则
- 考虑智能指针模式
- 或使用arena在编译阶段结束时统一释放
